# 後端分層重構收斂報告（2026-04-02）

## 4. 分層規範（當前準則）

- Blueprint：
  - 只做 input parsing、auth decorator、HTTP status mapping。
  - 不直接承載商業判斷與跨模型流程。

- Service：
  - 聚焦業務規則、權限判斷、跨資源協調與交易邊界。
  - 例外透過 OperationError 類型映射為 API 錯誤。

- Repository：
  - 集中 model query/filter/order/join。
  - 提供可重用 helper，避免 Service 散落直接 ORM 查詢。

## 5. 後續開發守則

- 新增 API 時，優先先設計 service 函式，再補 repository helper，不把 query 寫回 blueprint。
- 若 service 出現重複查詢片段，立即下沉到 repository。
- 每輪重構完成後，至少跑一次全量 `pytest tests -q` 再合併。

## 6. 風險與邊界

- AI 摘要目前以 Gemini 為主，切換模型時應只更動摘要實作函式，避免擴散到 blueprint 與 repository。
- task service 雖已顯著收斂，但仍建議持續保持「單一入口資料存取」原則，避免後續回滲 direct query。
