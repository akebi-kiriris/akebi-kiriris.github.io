# 專案封箱與下一步建議（2026-05-08）

> 目的：在不新增產品功能的前提下，為目前專案做收尾工程化，並提煉可帶去新專案的資產。

---

## 為什麼現在適合做「封箱級工程化」

目前主線功能已可用，且已完成多輪重構與契約收斂。  
這個階段再加功能的邊際效益較低，優先做穩定化、可維運化與可移植化，對後續專案延伸與新專案啟動更有價值。

---

## 建議優先順序（不新增功能）

## 1) 封版穩定化（最高優先）

- 針對主流程建立一份固定回歸清單：
  - 登入/登出
  - 專案 CRUD
  - 任務 CRUD + 指派 + 子任務
  - 知識庫文件上傳/刪除/reindex
  - RAG 規劃請求與來源顯示
- 輸出 `Release Checklist` 文件（可供每次發版照表執行）

**目標**：任何人（含未來的自己）都能快速判斷版本是否穩定。

## 2) 模組邊界最後收斂

- 對仍偏大的檔案做最後一輪責任拆分（例如 `TimelineDetailDialog`）
- 原則：不改需求、不改行為，只做責任切割與命名清晰化

**目標**：降低未來維護成本與修改風險。

## 3) 觀測性與除錯體驗

- 統一關鍵 log 欄位：`request_id`、`error_code`、`route`、`service`
- 錯誤回應延續目前 envelope（`error`/`error_code`/`error_details`）
- 補一頁「常見故障定位流程」小 runbook

**目標**：出問題可在短時間內定位，不依賴個人記憶。

## 4) 文件產品化

- 收斂為三份核心文件：
  1. 架構與責任邊界
  2. API 契約（含錯誤碼策略）
  3. 本地開發/測試/部署 runbook

**目標**：專案可交接、可展示、可快速重啟。

## 5) 測試策略固定化

- 定義固定測試分層：
  - `smoke`：最小主流程
  - `domain`：task/timeline/knowledge
  - `full`：全量回歸
- 將每次改動對應到「至少要跑哪一層」

**目標**：避免每次都臨時決定測試範圍。

---

## 可帶去新專案的高價值資產

你提出的方向非常好：  
**「函數描述 + 參數型別，做成 tool 給模型調用」**，這是強工程化、可重用的能力。

建議最小規格：

1. `tool manifest`：
   - `name`
   - `description`
   - `params schema`
2. 型別/Schema 自動導出（Pydantic 或 TypeScript）
3. Tool guard：
   - 權限檢查
   - 參數驗證
   - timeout
   - 錯誤碼映射（延續 `error_code`）

**收益**：這會變成可複製到新專案的底層能力，而不是一次性功能。

---

## 建議結論

現階段最值得做的是「封箱級工程化」：  
先把穩定性、可維運性、可觀測性收斂完成，再帶著一套成熟的工程骨架開新專案。
