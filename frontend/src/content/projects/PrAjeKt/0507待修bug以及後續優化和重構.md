# 待修bug以及後續優化和重構

A. 抽離驗證邏輯 (Validator)
目前的 if 'priority' in data 等判斷佔據了大部分篇幅。在 Clean Architecture 中，會傾向將「資料格式驗證」交給專門的 Schema（如 Marshmallow 或 Pydantic）。

優點：Service 層會縮短 60%，讓你一眼就看到業務邏輯的核心（例如任務依賴關係的處理）。

B. 封裝狀態變更 (Domain Method)
關於 status 的變更邏輯：

Python
if 'status' in data:
    next_status = data['status']
    task.status = next_status
    # ... 一連串的連動 ...
這其實是「任務」這個物件本身的行為。你可以考慮在 Task Model 裡面寫一個方法：

Python
### 在 Model 裡定義
def change_status(self, new_status):
    self.status = new_status
    self.completed = (new_status == 'completed')
    # ...
這樣 Service 層只需要調用 task.change_status(data['status'])，這就是 「胖 Model, 瘦 Service」 的設計模式。

C. 事務控制 (Transaction Context)
你目前在 Service 層結尾寫了 db.session.commit()。
在更大型的架構中，有時一個 Service 會調用另一個 Service，這時 commit 的時機點會變得很尷尬（可能 A 成功了但 B 失敗，導致資料不一致）。

建議：通常會把 commit 移到最外層（Controller 或 Middleware），或是使用 Unit of Work 模式 來統一管理。

---

在 Clean Architecture 或領域驅動設計（DDD）的觀點中，驗證其實分為兩種，你的疑惑來自於這兩者的邊界：

輸入驗證 (Input Validation)：檢查格式、型別、長度。例如：「優先級是不是 1~3 的數字？」、「Email 格式對嗎？」

業務驗證 (Business Validation)：檢查邏輯與狀態。例如：「任務相依性會不會造成循環？」、「該使用者是否有權限修改這項任務？」

1. Pydantic 應該放在 Blueprint 層 (輸入驗證)
通常建議將 Pydantic 放在 Blueprint (Controller) 層。理由如下：

提早攔截 (Fail Fast)：
如果使用者傳進來的資料格式根本不對（例如 priority 傳了字串 "High"），我們在 Blueprint 層就該直接擋掉並回傳 400 Bad Request。完全不需要驚動 Service 層，更不需要浪費資料庫資源去查 _find_active_task_or_404。

Service 層的純淨度：
Service 層應該假設它拿到的資料已經是「合法格式」的。它只負責處理最核心、最難的業務邏輯（例如你寫的 validate_dependency_task_ids）。

2. Service 層負責「業務驗證」
雖然 Pydantic 擋掉了格式問題，但有些事情它擋不掉，這些就該留在 Service：

跨表檢查：檢查 timeline_id 是否真的存在且屬於該專案。

狀態轉移：檢查任務是否可以從「已完成」改回「進行中」。

相依性檢查：這就是你程式碼中 validate_dependency_task_ids 做的事，這是 Service 層的職責，因為它涉及了資料庫的狀態查詢。

3. 實際的代碼結構看起來會像這樣
你的專案結構可以調整為：

blueprints/task_bp.py (負責接客與安檢)：

拿到 JSON。

用 TaskUpdateSchema 驗證格式。

若失敗，直接回傳 400。

若成功，呼叫 Service。

services/task_service.py (負責執行與業務判斷)：

找到 Task 物件。

檢查業務邏輯（例如權限、任務相依性）。

執行更新。

```python
def validate_dependency_task_ids(timeline_id, depends_on_task_ids, current_task_id=None):
    if not depends_on_task_ids:
        return

    if timeline_id in (None, ''):
        raise TaskOperationError('設定 depends_on_task_ids 前需先指定 timeline_id', 400)

    if current_task_id is not None and current_task_id in depends_on_task_ids:
        raise TaskOperationError('depends_on_task_ids 不可包含自己', 400)

    active_task_ids = {
        task.task_id
        for task in Task.query.filter(
            Task.timeline_id == timeline_id,
            Task.deleted_at.is_(None),
        ).all()
    }
```
