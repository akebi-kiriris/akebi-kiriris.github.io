## 1. 前言與目的

現代 Web 應用的穩定性高度依賴自動化測試。良好的測試能夠：

- 使重構更安全（refactor-with-confidence），減少回歸。
- 作為設計文件，強迫程式碼有明確的邊界（service vs blueprint/controller）。
- 在 CI 中自動驗證部署安全性。

本文件以專案實務出發，詳述如何在後端（Flask + SQLAlchemy）與前端（Vite）建立可靠、可維護的測試體系，並包含面試會問的重點整理。

## 1.1 什麼是自動化測試？（完全初級者版本）

**定義**：自動化測試是用程式碼來驗證你的應用程式是否正確運作，而不是靠人手手動檢查。

**簡單例子**：
假設你寫了一個函式 `double(x)` 來計算 x 的兩倍。

```python
def double(x):
    return x * 2
```

**手動測試**（不好的做法）：
每次修改 double 函式後，你都得在 Python 命令列手動執行：
```
>>> double(5)
10
>>> double(0)
0
>>> double(-3)
-6
```

問題：
- 容易遺漏某些 case（例如忘記測試負數）
- 每次程式改動都要重複手動測試
- 很快就會放棄，因為太麻煩

**自動化測試**（好的做法）：
寫一次測試程式碼，然後**自動執行無數次**：

```python
def test_double_positive():
    assert double(5) == 10

def test_double_zero():
    assert double(0) == 0

def test_double_negative():
    assert double(-3) == -6
```

執行測試：
```bash
pytest test_double.py
# 輸出
# test_double.py::test_double_positive PASSED
# test_double.py::test_double_zero PASSED
# test_double.py::test_double_negative PASSED
# 3 passed in 0.05s
```

現在，每次修改 double 函式後，只需執行一個命令，自動檢查所有 case 是否仍然正確。

**自動化測試 vs 手動測試**

| 方面 | 手動測試 | 自動化測試 |
|------|---------|---------|
| 首次投入 | 很快 | 稍慢（寫測試代碼需時間） |
| 重複成本 | 每次都要手動執行，費時 | 一次執行，輸出結果（秒級） |
| 覆蓋度 | 容易遺漏 | 可以設計全面的 case |
| 長期維護 | 難以維護，bug 易重複 | 易於維護，bug 一旦修復就不會回來 |
| 人工錯誤 | 高（容易看錯結果） | 低（程式自動判斷） |

**自動化測試 vs 程式碼審查（Code Review）**

- **程式碼審查**：檢查代碼的**結構、邏輯、命名、複雜度**是否良好
- **自動化測試**：檢查代碼**執行結果**是否正確

兩者互補，不能替代。

**自動化測試的三大好處**

1. **提早發現 bug**
   ```python
   # 舊函式
   def calculate_age(birth_year):
       return 2024 - birth_year
   
   # 寫了測試
   def test_age():
       assert calculate_age(2000) == 24
   
   # 三個月後，你改了函式但忘記了 case
   def calculate_age(birth_year):
       return 2025 - birth_year  # bug！
   
   # 執行測試：FAILED！被當場抓到，而不是上線後才發現
   ```

2. **安心重構**
   ```python
   # 原本的實現
   def get_even_numbers(nums):
       result = []
       for n in nums:
           if n % 2 == 0:
               result.append(n)
       return result
   
   # 有測試確保它正確
   def test_get_even_numbers():
       assert get_even_numbers([1, 2, 3, 4]) == [2, 4]
   
   # 現在你可以大膽重構為更簡潔的版本
   def get_even_numbers(nums):
       return [n for n in nums if n % 2 == 0]
   
   # 執行測試：仍然 PASSED，說明重構是安全的
   ```

3. **作為文件**
   ```python
   def test_user_login_with_correct_credentials():
       """用戶用正確的用戶名和密碼登入，應該成功。"""
       result = login(username='alice', password='secret123')
       assert result['success'] == True
       assert result['user_id'] is not None
   
   # 三個月後，新成員看到這個測試，就知道 login 函式的預期行為
   ```

**測試 vs 實際應用的關係**

```
實際應用代碼（Application Code）
↓ 被驗證 ↓
測試代碼（Test Code）
↑ 驗證 ↑
實際應用代碼
```

測試程式碼會**呼叫**應用程式代碼，並驗證其行為。

**為什麼在面試時會問測試？**

1. **展示工程紀律**：有寫測試的人通常代碼品質更高
2. **減少線上事故**：自動化測試能防止明顯的 bug 進線上
3. **團隊協作**：有測試的代碼更容易讓別人維護
4. **快速回應**：當需要緊急修復時，測試能確保不會引入新 bug

## 2. 測試原則與分類

- 單一責任：每個測試應該只驗證一個行為。
- 可重現：測試結果不應受環境非 determinism 影響。
- 隔離：單元測試應避免依賴外部服務；integration / e2e 才測整合面。
- 速度優先：將速度快的測試放在 CI pipeline 前段，慢且昂貴的 E2E 放在後段或夜間任務。

分類：
- Unit：單一 function/class 行為，不接觸網路、資料庫（用 mock）。
- Integration / Service：測試服務層邏輯，可能使用真實 DB（測試 DB，例如 in-memory SQLite）。
- Controller / Route：透過 Flask test client 呼叫 HTTP endpoints，驗證 request/response 與驗證、權限映射。
- E2E：整個系統（前端 + 後端 + DB 或 mock 後端）驗證使用者流程。

## 2.1 pytest 是什麼？（測試框架初級解釋）

**定義**：pytest 是 Python 最流行的**測試框架**，用來組織、執行、報告測試結果。

**類比**：如果測試代碼是「檢查清單」，pytest 就是「負責逐一檢查並統計結果的主管」。

**pytest 的核心職責**：
1. **自動發現測試**：找到所有以 `test_` 開頭的函式或類別
2. **執行測試**：按順序執行，捕捉例外與失敗
3. **報告結果**：用彩色輸出顯示通過/失敗
4. **提供工具**：例如斷言、fixtures、markers 等

**最簡單的使用方式**：

假設你有這個測試檔案：

```python
# test_calculator.py
def test_add():
    assert 2 + 2 == 4

def test_subtract():
    assert 5 - 3 == 2
```

執行：
```bash
cd 你的項目目錄
pytest test_calculator.py
```

輸出：
```
test_calculator.py::test_add PASSED
test_calculator.py::test_subtract PASSED
============= 2 passed in 0.05s =============
```

**pytest 的約定（Convention）**：

| 規則 | 說明 |
|------|------|
| 檔案名稱 | 以 `test_` 開頭，例如 `test_user.py` |
| 測試函式 | 以 `test_` 開頭，例如 `def test_login_success():` |
| 測試類別 | 以 `Test` 開頭，例如 `class TestUser:` |
| 斷言 | 使用 Python 原生的 `assert` 關鍵字 |

**pytest 常用命令**：

```bash
pytest                          # 執行所有測試
pytest test_user.py             # 執行特定檔案
pytest -v                       # 詳細輸出（verbose）
pytest -x                       # 第一個失敗後停止
pytest -s                       # 顯示 print 的內容
pytest --tb=short               # 簡短的失敗堆棧跟蹤
pytest -k "test_login"          # 只執行名稱含 test_login 的測試
pytest -m "slow"                # 執行標記為 slow 的測試
pytest --collect-only           # 列出所有測試（不執行）
```

**為什麼不用 unittest？**

Python 還有內建的 `unittest` 框架，但 pytest 更簡潔：
- `unittest` 要求寫在類別中，使用 `self.assertEqual()` 等方法（冗長）
- `pytest` 可以寫簡單的函式，使用 Python 原生 `assert`（簡潔）
- `pytest` 的 fixtures 比 unittest 的 setUp/tearDown 更靈活

## 2.2 什麼是 Fixture？（測試的「準備階段」助手）

**定義**：Fixture 是一段**可重複使用的準備代碼**，為多個測試提供共同的資源（例如資料、連接、臨時文件）。

**類比**：如果測試是「烤蛋糕」，fixture 就是「提前準備好的原料和工具」。

**為什麼需要 Fixture？**

假設你有多個測試都需要一個使用者對象：

```python
# 不好的做法：重複代碼
def test_user_login():
    user = User(name='Alice', email='alice@ex.com', password='secret')
    assert user.name == 'Alice'

def test_user_email():
    user = User(name='Alice', email='alice@ex.com', password='secret')
    assert user.email == 'alice@ex.com'

def test_user_validate():
    user = User(name='Alice', email='alice@ex.com', password='secret')
    assert user.is_valid()
```

問題：User 建立代碼重複了 3 次，若要修改（例如加 username 欄位），要改 3 個地方。

**使用 Fixture 解決**：

```python
import pytest

@pytest.fixture
def sample_user():
    """提供一個樣本使用者。"""
    return User(name='Alice', email='alice@ex.com', password='secret')

def test_user_login(sample_user):
    assert sample_user.name == 'Alice'

def test_user_email(sample_user):
    assert sample_user.email == 'alice@ex.com'

def test_user_validate(sample_user):
    assert sample_user.is_valid()
```

現在，如果要改使用者資料，只需改一個地方。

**Fixture 的生命週期（Scope）**：

Fixture 可以決定在什麼級別「準備」和「清理」資源：

```python
@pytest.fixture(scope="function")  # 預設：每個測試函式執行前後
def db_connection():
    conn = connect_to_db()
    yield conn
    conn.close()

@pytest.fixture(scope="module")  # 整個模組內共享一次
def expensive_resource():
    resource = create_expensive_resource()
    yield resource
    cleanup(resource)
```

**Fixture 與 yield 的含義**：

```python
@pytest.fixture
def app():
    """設定一個 Flask app，測試後清理。"""
    app = Flask(__name__)
    
    # Setup（準備）
    app.config['TESTING'] = True
    db.init_app(app)
    db.create_all()
    
    # yield：給測試使用
    yield app
    
    # Teardown（清理）
    db.drop_all()
    db.session.remove()
```

這個 fixture：
1. 建立 app 並初始化資料庫（setup）
2. 給測試使用（yield）
3. 清理資料庫（cleanup）

**常見的 Fixture 模式**（在本專案中會看到）：

```python
@pytest.fixture
def app():
    """Flask app 實例。"""
    app = Flask(__name__)
    # ... 初始化 ...
    yield app

@pytest.fixture
def client(app):
    """依賴於 app fixture 的 test client。"""
    return app.test_client()

@pytest.fixture
def sample_user(app):
    """資料庫中的樣本使用者。"""
    user = User(name='Test', email='test@ex.com', password='hashed')
    db.session.add(user)
    db.session.commit()
    return user
```

## 2.5 測試的基本結構與寫法（面試入門）

**Q：測試怎麼寫？最基本的結構是什麼？**

答：每個測試遵循 **AAA 模式**（Arrange-Act-Assert）：

```
1. Arrange（準備）：建立測試資料、fixture、mocks
2. Act（執行）：呼叫要測試的函式或方法
3. Assert（驗證）：檢查結果是否符合預期
```

**例 1：測試一個簡單的計算函式**

```python
# 被測試的函式
def double(x):
    return x * 2

# 測試函式
def test_double():
    # Arrange
    input_value = 5
    expected_output = 10
    
    # Act
    result = double(input_value)
    
    # Assert
    assert result == expected_output
```

pytest 會自動發現並執行所有以 `test_` 開頭的函式或以 `Test` 開頭的類別。

**例 2：測試一個可能拋出例外的函式**

```python
import pytest

def divide(a, b):
    if b == 0:
        raise ValueError("Cannot divide by zero")
    return a / b

def test_divide_success():
    # Arrange & Act & Assert
    assert divide(10, 2) == 5

def test_divide_by_zero_raises_error():
    # Arrange
    with pytest.raises(ValueError) as exc_info:
        # Act
        divide(10, 0)
    
    # Assert
    assert "Cannot divide by zero" in str(exc_info.value)
```

**例 3：測試一個涉及資料庫的函式**

```python
from models import db, User

def test_create_user_in_database(app):  # app 是 fixture
    """測試使用者建立並存入資料庫。"""
    # Arrange
    user_data = {
        'name': 'Alice',
        'email': 'alice@example.com',
        'username': 'alice_user',
    }
    
    # Act
    user = User(**user_data, password='hashed')
    db.session.add(user)
    db.session.commit()
    
    # Assert
    fetched_user = db.session.query(User).filter_by(email='alice@example.com').first()
    assert fetched_user is not None
    assert fetched_user.name == 'Alice'
    assert fetched_user.id is not None  # commit 後應有 id
```

**常用的斷言方法（Assertions）：**

```python
# 基本斷言
assert x == y                  # 相等
assert x != y                  # 不相等
assert x > y                   # 大於
assert x < y                   # 小於
assert x in [a, b, c]          # 包含
assert not x                   # 為 False 或 None

# 物件斷言
assert isinstance(obj, ClassName)    # 型別檢查
assert hasattr(obj, 'attribute')     # 是否有屬性
assert obj is not None               # 非 None

# 字符串斷言
assert 'substring' in 'long string'  # 子字符串檢查
assert 'error' in error_message.lower()  # 不區分大小寫檢查

# 列表 / 字典斷言
assert len(my_list) == 3
assert task['title'] == 'Expected Title'
assert any(t.id == expected_id for t in tasks)  # 任何一個符合

# 異常斷言（已示範）
with pytest.raises(ValueError):
    function_that_should_raise()
```

**例 4：測試 HTTP 端點（使用 Flask test client）**

```python
def test_create_task_api(app):
    """測試 POST /api/tasks 端點。"""
    # Arrange
    client = app.test_client()
    user = User(name='Bob', email='bob@ex.com', username='bob', password='h')
    db.session.add(user)
    db.session.commit()
    
    token = create_access_token(identity=user.id)  # 假設有此函式
    
    payload = {
        'title': 'New Task',
        'description': 'Test task',
        'priority': 2,
    }
    headers = {'Authorization': f'Bearer {token}'}
    
    # Act
    response = client.post('/api/tasks', json=payload, headers=headers)
    
    # Assert
    assert response.status_code == 201
    assert response.json['title'] == 'New Task'
    assert response.json['task_id'] is not None
    
    # 額外驗證：資料庫中是否真的建立了
    from models import Task
    task = db.session.query(Task).filter_by(id=response.json['task_id']).first()
    assert task is not None
```

**測試的命名約定：**

```python
# ✅ 好的命名
def test_create_task_with_valid_data_returns_201()
def test_create_task_without_token_returns_401()
def test_create_task_with_duplicate_title_raises_error()
def test_list_tasks_returns_only_current_user_tasks()

# ❌ 不好的命名
def test_task()                    # 太模糊
def test_creates()                 # 不明確
def test_1()                       # 沒有意義
```

好的測試名稱應該清楚描述「什麼條件下」發生了「什麼行為」，並且「預期什麼結果」。

**pytest 標記（Markers）：組織與過濾測試**

```python
import pytest

@pytest.mark.slow
def test_fetch_large_dataset():
    """此測試較慢，通常在整合測試時執行。"""
    pass

@pytest.mark.integration
def test_database_operation():
    """整合測試，需要真實資料庫。"""
    pass

@pytest.mark.skip(reason="功能尚未實作")
def test_future_feature():
    """暫時跳過此測試。"""
    pass

@pytest.mark.xfail(reason="已知 bug，稍後修復")
def test_known_bug():
    """標記為預期失敗，但若通過也不會導致 CI 失敗。"""
    pass
```

執行時可過濾：
```bash
pytest -m "not slow"              # 執行除了 slow 外的所有測試
pytest -m "integration"           # 只執行 integration 標記的測試
pytest -v --co                    # 列出所有測試（不執行）
```


## 2.6 什麼是 ORM？（SQLAlchemy 初級概念）

**定義**：ORM（Object-Relational Mapping）是一種技術，讓你用**物件**而不是 SQL 來操作資料庫。

**類比**：
- **沒有 ORM（直接寫 SQL）**：像用外國語言寫信，需要知道 SQL 語法
- **有 ORM（SQLAlchemy）**：像用自己的母語寫信，ORM 替你翻譯成 SQL

**簡單例子**

假設資料庫中有一個 `users` 表：
```sql
-- 表的結構
CREATE TABLE users (
    id INTEGER PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100),
    password VARCHAR(255)
);
```

**傳統 SQL 操作**（複雜且容易出錯）：
```python
import sqlite3

conn = sqlite3.connect('database.db')
cursor = conn.cursor()

# 新增使用者
cursor.execute(
    "INSERT INTO users (name, email, password) VALUES (?, ?, ?)",
    ('Alice', 'alice@ex.com', 'hashed')
)
conn.commit()

# 查詢使用者
cursor.execute("SELECT * FROM users WHERE email = ?", ('alice@ex.com',))
user = cursor.fetchone()  # 返回 tuple，例如 (1, 'Alice', 'alice@ex.com', 'hashed')
print(user[1])  # 訪問名字，需要記住索引 1

# 刪除使用者
cursor.execute("DELETE FROM users WHERE id = ?", (user[0],))
conn.commit()
```

問題：
- 需要寫 SQL 語句
- 結果是 tuple，無法直觀訪問欄位（需要用索引）
- 容易產生 SQL injection 漏洞

**使用 ORM（SQLAlchemy）**（簡潔且安全）：
```python
from flask_sqlalchemy import SQLAlchemy

db = SQLAlchemy()

# 定義 User 模型（對應資料庫表）
class User(db.Model):
    __tablename__ = 'users'
    
    id = db.Column(db.Integer, primary_key=True)
    name = db.Column(db.String(100), nullable=False)
    email = db.Column(db.String(100), unique=True, nullable=False)
    password = db.Column(db.String(255), nullable=False)

# 現在使用物件操作，而不是 SQL
# 新增使用者
user = User(name='Alice', email='alice@ex.com', password='hashed')
db.session.add(user)
db.session.commit()

# 查詢使用者
user = db.session.query(User).filter_by(email='alice@ex.com').first()
print(user.name)  # 直觀訪問 name 屬性
print(user.email)

# 修改使用者
user.name = 'Alice Updated'
db.session.commit()

# 刪除使用者
db.session.delete(user)
db.session.commit()
```

好處：
- 程式碼可讀性高，接近自然語言
- 自動防止 SQL injection
- Python 物件 → 自動轉為 SQL（ORM 負責翻譯）
- 資料庫結果 → 自動轉為 Python 物件

**ORM 的核心概念**

1. **模型（Model）**：Python 類別，對應資料庫表
```python
class Task(db.Model):
    __tablename__ = 'tasks'
    
    id = db.Column(db.Integer, primary_key=True)
    title = db.Column(db.String(200), nullable=False)
    description = db.Column(db.Text)
    user_id = db.Column(db.Integer, db.ForeignKey('users.id'))
    created_at = db.Column(db.DateTime, default=datetime.utcnow)
```

2. **Session**：資料庫連接與交易管理器
```python
db.session.add(obj)         # 添加物件
db.session.commit()          # 提交改變（寫入資料庫）
db.session.rollback()        # 撤銷改變
db.session.query(Model)      # 查詢物件
```

3. **查詢**：使用 Python 方法而不是 SQL
```python
# SQL: SELECT * FROM users WHERE name = 'Alice'
users = db.session.query(User).filter_by(name='Alice').all()

# SQL: SELECT * FROM tasks WHERE user_id = 1 ORDER BY created_at DESC LIMIT 10
tasks = db.session.query(Task).filter_by(user_id=1).order_by(Task.created_at.desc()).limit(10).all()

# SQL: SELECT * FROM tasks WHERE title LIKE '%bug%'
tasks = db.session.query(Task).filter(Task.title.contains('bug')).all()
```

**在測試中的應用**

```python
def test_create_user(app):
    """使用 ORM 建立並驗證使用者。"""
    # 建立使用者物件（在記憶體中）
    user = User(name='Alice', email='alice@ex.com', password='hashed')
    
    # 添加到 session（標記為待提交）
    db.session.add(user)
    
    # 提交到資料庫（分配 id）
    db.session.commit()
    
    # 驗證
    assert user.id is not None
    
    # 查詢驗證（從資料庫中取出）
    fetched = db.session.query(User).filter_by(email='alice@ex.com').first()
    assert fetched is not None
    assert fetched.name == 'Alice'
```

**重點總結**：
- ORM 是 Python 物件與資料庫表的橋樑
- SQLAlchemy 是 Python 最流行的 ORM
- 使用 ORM 使代碼更易讀、更安全
- 在測試中，ORM 讓資料準備和驗證都變得簡單

## 3. 後端測試環境與約定（以本專案為例）

### 3.1 `tests/conftest.py` 的角色

在本專案（参見 `tests/conftest.py`），`app` fixture 建立一個 Flask app 用於測試：

- 設定 `TESTING = True` 以啟用 Flask 的測試模式。
- 設定 `SQLALCHEMY_DATABASE_URI = "sqlite:///:memory:"`，表示使用記憶體 SQLite。
- 呼叫 `db.init_app(app)` 初始化 SQLAlchemy。
- 在 `with app.app_context(): db.create_all(); yield app; db.session.remove(); db.drop_all()` 中建立與清理 schema。

這個 fixture 保證每次 `app` 被使用時會建立一個乾淨的資料庫環境，測試結束時清除。

### 3.2 為何使用 `sqlite:///:memory:`？

優點：
- 非常快速（I/O 負擔小），適合大量測試。
- 每個測試或 fixture create/drop 可以達到隔離，避免測試互相污染。

限制：
- SQLite 與 Postgres 在 SQL dialect、交易、約束行為上有差異（例如：某些 SQL 功能、DDL 行為、時間欄位精度）。
- 若程式使用 Postgres 特有功能（如 JSONB、全文檢索），單純的 SQLite 測試可能無法覆蓋。

實務建議：
- 以 SQLite in-memory 作為快速回歸測試；在 CI 或每日整合測試再跑一次使用真實 Postgres 的整合測試。

### 3.3 為何測試中會看到 `db.session.add()` 與 `db.session.commit()`？

- 在 service 層測試中，當你要建立測試資料（fixtures）時，通常會 `db.session.add(obj)` 並 `db.session.commit()`：
  - `commit()` 會將物件寫入資料庫並分配主鍵（PK），若 service 函式內部根據 PK 或已提交狀態執行查詢或關聯（例如建立子物件時使用父物件的 id），必須先 commit。
  - 若是 transaction-scoped 的邏輯（例如 service 在同一個 transaction 中讀寫），可以用 `flush()` 或在同一 session 中操作，但 commit 確保資料持久並可由新的 session/查詢看到。

- 設計要點：
  - Service 層測試（測試純粹業務邏輯）應該直接呼叫 service function 並在測試內備妥必要資料（commit 後）以模擬真實工作流。
  - Controller 層測試（透過 HTTP 呼叫）則依賴 app fixture 的資料庫；若測試流程需先 log in，則測試會先建立 user, commit，然後呼叫登入 endpoint 取得 token，再進行需要授權的請求。

### 4.2.1 Flask Test Client 完整教學

**Q：什麼是 Flask test client？**

Flask test client 是一個模擬 HTTP 客戶端的物件，用於在不啟動真實 Web 伺服器的情況下，直接向 Flask 應用發送 HTTP 請求。

```python
from flask import Flask

app = Flask(__name__)
client = app.test_client()  # 建立 test client
```

**Flask test client 的主要方法與參數詳解：**

#### 1. POST 請求（建立資源）
```python
response = client.post(
    '/api/tasks',                           # 路徑
    json={'title': 'Task', 'priority': 2},  # JSON body（自動序列化）
    headers={'Authorization': 'Bearer token'}  # HTTP headers
)
```

參數說明：
- `'/api/tasks'` - 端點路徑
- `json=<dict>` - 傳送 JSON 資料（Content-Type 自動設為 application/json）
- `data=<dict>` 或 `data=<str>` - 傳送表單資料或文本
- `headers=<dict>` - HTTP headers（例如認證、Content-Type）
- `query_string=<dict>` 或 URL 中直接帶 `?key=value` - URL query 參數

返回值：
```python
response.status_code      # HTTP 狀態碼，整數（例如 201）
response.json             # 解析後的 JSON 回應（字典）
response.data             # 原始響應內容（bytes）
response.text             # 響應內容（字符串）
response.headers          # 響應 headers（字典）
response.content_type     # Content-Type header

# 例如
assert response.status_code == 201
assert response.json['task_id'] == 1
```

#### 2. GET 請求（取得資源）
```python
response = client.get(
    '/api/tasks/1',
    headers={'Authorization': 'Bearer token'}
)

assert response.status_code == 200
task = response.json  # 整個 response 是 task 物件
```

參數說明：
- 類似 POST，但通常不帶 `json` / `data`
- 若需傳參數，使用 query string：`'/api/tasks?page=1&limit=10'` 或 `query_string={'page': 1, 'limit': 10}`

#### 3. PUT 請求（更新整個資源）
```python
response = client.put(
    '/api/tasks/1',

```

```python
response = client.patch(
```

```python
def test_complete_task_workflow(app):
    """完整工作流測試，展示所有 HTTP 方法。"""
    client = app.test_client()
    
    # 1. 先建立使用者（在資料庫中）
    from models import db, User
    user = User(name='Alice', username='alice', email='alice@ex.com', password='hashed')
    db.session.add(user)
    db.session.commit()
    
    # 2. 登入以取得 token
    login_res = client.post('/api/auth/login', json={
        'username': 'alice',
        'password': 'password',  # 假設密碼驗證成功
    })
    assert login_res.status_code == 200
    token = login_res.json['access_token']
    headers = {'Authorization': f'Bearer {token}'}
    
    # 3. POST - 建立任務
    create_res = client.post('/api/tasks', json={
        'title': 'Important Task',
        'priority': 1,
    }, headers=headers)
    assert create_res.status_code == 201
    task_id = create_res.json['task_id']
    
    # 4. GET - 查詢任務
    get_res = client.get(f'/api/tasks/{task_id}', headers=headers)
    assert get_res.status_code == 200
    assert get_res.json['title'] == 'Important Task'
    
    # 5. PUT - 更新任務
    update_res = client.put(f'/api/tasks/{task_id}', json={
        'title': 'Updated Task',
        'priority': 2,
    }, headers=headers)
    assert update_res.status_code == 200
    assert update_res.json['title'] == 'Updated Task'
    
    # 6. DELETE - 刪除任務
    delete_res = client.delete(f'/api/tasks/{task_id}', headers=headers)
    assert delete_res.status_code == 204
    
    # 7. 驗證已刪除（GET 應返回 404）
    verify_res = client.get(f'/api/tasks/{task_id}', headers=headers)
    assert verify_res.status_code == 404
```

**常見 HTTP 狀態碼及含義（面試常問）：**

| 狀態碼 | 含義 | 何時返回 |
|-------|------|--------|
| 200 | OK，請求成功 | GET / PUT / PATCH 成功 |
| 201 | Created，資源已建立 | POST 成功（新建立資源） |
| 204 | No Content，成功但無內容 | DELETE 成功、某些 PATCH 成功 |
| 400 | Bad Request，請求參數錯誤 | JSON 格式錯誤、必填欄位缺失、驗證失敗 |
| 401 | Unauthorized，需要認證 | 無 token 或 token 無效 |
| 403 | Forbidden，無權限 | 有 token 但無權訪問此資源（例如非所有者刪除） |
| 404 | Not Found，資源不存在 | GET 不存在的任務、DELETE 已刪除的資源 |
| 409 | Conflict，衝突 | 邏輯衝突（例如重複建立、修改時版本不符） |
| 429 | Too Many Requests，速率限制 | 請求過於頻繁 |
| 500 | Internal Server Error，伺服器錯誤 | 未預期的例外 |

**面試常見問題 1：如何測試 401 vs 403？**

```python
# 401：無 token
res = client.post('/api/tasks', json={'title': 'X'})  # 不帶 headers
assert res.status_code == 401

# 403：有 token 但無權限
user1 = User(...)
user2 = User(...)
db.session.add_all([user1, user2])
db.session.commit()

token1 = create_access_token(identity=user1.id)
headers = {'Authorization': f'Bearer {token1}'}

# User 1 建立任務
task_id = ...

# User 2 嘗試刪除 User 1 的任務

assert res.status_code == 403  # 有 token，但無權限
```

**面試常見問題 2：如何測試請求驗證（400）？**

```python
# 缺少必填欄位
res = client.post('/api/tasks', json={}, headers=headers)
assert res.status_code == 400
assert 'title' in res.json['error'].lower()

# 無效的資料型別
res = client.post('/api/tasks', json={'title': 123}, headers=headers)  # title 應是字符串
assert res.status_code == 400

# 無效的 priority 值（超出範圍）

```

### 4.3 Fixture 設計策略

- 小型專案：在 `conftest.py` 提供一個 `app` fixture。
- 若多個測試需要同一組資料（例如 sample users），可建立 `session` 或 `module` scope 的 fixtures 來提高速度，但要注意隔離性。

### 4.3.1 什麼是 Mock？為什麼需要？（初級解釋）

**定義**：Mock 是一個**假的對象**，用來替代真實的外部服務（例如資料庫、API、第三方服務），使測試能夠獨立執行。

**類比**：如果你要測試「餐廳點餐系統」，但廚房還沒準備好，你會用一個「假廚房」來測試，以確保點餐邏輯正確。真實廚房準備好後，再進行整合測試。

**為什麼需要 Mock？**

假設你有一個函式會呼叫 OpenAI API：

```python
def summarize_text(text):
    """使用 OpenAI 總結文字。"""
    response = openai.ChatCompletion.create(
        model="gpt-4",
        messages=[{"role": "user", "content": f"Summarize: {text}"}]
    )
    return response['choices'][0]['message']['content']
```

**問題**（沒有 mock 時）：
1. **緩慢**：每次測試都要呼叫真實 API，往返延遲 1-5 秒
2. **不穩定**：網路可能中斷、API 可能故障
3. **昂貴**：每個測試都消耗 API quota，可能產生費用
4. **難以測試邊界情況**：無法輕易模擬「API 返回錯誤」的情況
5. **非決定性**：同樣的測試可能由於 API 改動而失敗

**解決方案**（使用 Mock）：

```python
def test_summarize_text(monkeypatch):
    """測試總結函式，使用 mock 替代 API。"""
    
    # 定義假的 API 回應
    def fake_openai_call(*args, **kwargs):
        class FakeResponse:
            def __getitem__(self, key):
                if key == 'choices':
                    return [{'message': {'content': 'Fake summary'}}]
        return FakeResponse()
    
    # 用假函式替換真實的 OpenAI API
    monkeypatch.setattr('openai.ChatCompletion.create', fake_openai_call)
    
    # 現在測試會使用假的 API
    result = summarize_text("Very long text...")
    
    assert result == 'Fake summary'
    # 測試快速、穩定、不消耗 quota
```

**Mock 帶來的好處**：
1. **快速**：假 API 立即返回，不需等待
2. **穩定**：不依賴網路或外部服務
3. **便宜**：不消耗 API quota 或費用
4. **完整覆蓋**：可以測試 API 返回錯誤的情況
5. **可預測**：每次都返回相同結果

**Monkeypatch 是什麼？**

Monkeypatch 是 pytest 提供的工具，用來**暫時替換**（patch）函式或物件。

```python
def test_example(monkeypatch):
    """展示 monkeypatch 的三種常見用法。"""
    
    # 1. 替換函式
    def fake_function():
        return "fake result"
    
    monkeypatch.setattr('module.function', fake_function)
    # 現在 module.function 會返回 "fake result"
    
    # 2. 修改環境變數
    monkeypatch.setenv('API_KEY', 'fake-key-for-testing')
    # 現在 os.getenv('API_KEY') 返回 'fake-key-for-testing'
    
    # 3. 刪除環境變數
    monkeypatch.delenv('PRODUCTION_FLAG')
    # 現在 PRODUCTION_FLAG 不存在（測試只在開發環境中）
    
    # 測試結束後，所有改動自動撤銷
```

**重點**：Monkeypatch 的改動**只在測試期間有效**，測試結束後自動還原（不影響其他測試或全局狀態）。

**實用例子**：測試涉及外部 API 的代碼

```python
# 被測試的代碼
def get_user_from_api(user_id):
    """從外部 API 取得使用者資訊。"""
    response = requests.get(f'https://api.example.com/users/{user_id}')
    response.raise_for_status()
    return response.json()

# 測試代碼
def test_get_user_from_api(monkeypatch):
    """測試 get_user_from_api，mock 外部 API。"""
    
    # 定義假的 HTTP response
    def fake_get(url, **kwargs):
        class FakeResponse:
            status_code = 200
            def json(self):
                return {'id': 1, 'name': 'Alice', 'email': 'alice@ex.com'}
            def raise_for_status(self):
                pass
        return FakeResponse()
    
    # 替換 requests.get
    monkeypatch.setattr('requests.get', fake_get)
    
    # 現在測試使用假 API
    user = get_user_from_api(1)
    
    assert user['name'] == 'Alice'
    assert user['email'] == 'alice@ex.com'
```

**何時使用 Mock？**

- 呼叫外部 API（OpenAI、Google、AWS 等）
- 呼叫資料庫（除非你用測試 DB）
- 呼叫需要認證的服務（避免洩露密鑰）
- 呼叫會產生副作用的操作（寄信、刪除檔案等）
- 測試邊界情況（例如 API 超時、返回錯誤）

**何時 NOT 使用 Mock？**

- 測試純邏輯函式（不涉及外部服務）
- 整合測試（需要真實的子系統）
- 使用測試資料庫進行資料庫操作

### 4.4 Mock 與 Monkeypatch 實務用法

- 使用 `monkeypatch.setenv()` / `monkeypatch.delenv()` 臨時修改環境變數。
- 使用 `monkeypatch.setattr(target_module, 'function', fake_func)` 來替換外部 API 呼叫，例如替換 AI provider 的呼叫以回傳可控結果。
- 若是 HTTP client（requests / httpx），可使用 `responses` 或 `respx` 模擬遠端回應，或在前端使用 MSW 模擬。

### 4.5 檔案上傳 / 下載測試

- 使用 `werkzeug.datastructures.FileStorage` 或測試 client 的 `files={'file': (BytesIO(b"..."), 'name.png')}` 上傳檔案。
- 在檔案下載測試中檢查 `Content-Disposition` 與回傳的 bytes 是否符合預期。

範例：
```python
from io import BytesIO
resp = client.post('/api/tasks/1/files', headers=headers, data={'file': (BytesIO(b'hi'), 'a.txt')})
assert resp.status_code == 201
```

### 4.6 測試錯誤處理

- 在 service 層，定義 custom exception（例如 `TodoOperationError`）並包含 `status_code`。
- 在 blueprint 層統一捕捉這些錯誤並轉為適當 HTTP 回應。
- 測試要驗證 service raise 的錯誤最終對應到正確的 HTTP code（例如 400 / 404 / 403 / 500）。

### 6.1 SQLAlchemy ORM 與測試隔離策略

#### 6.1.1 Session 管理的重要性

SQLAlchemy 中的 `session` 是一個 unit-of-work 模式的實現。理解如何在測試中管理 session 對測試隱定性與效率至關重要。

**什麼是 Session？**
- Session 是 SQLAlchemy 與資料庫之間的窗口。
- 所有 ORM 物件（例如 `User`, `Task`）的生命週期都由所在的 session 管理。
- 一個 session 內的物件被跟蹤（tracked），修改會在 flush 或 commit 時寫入資料庫。

**Session 的主要操作：**
- `add(obj)`：在 session 中添加物件，標記為 pending。
- `flush()`：執行 SQL INSERT/UPDATE/DELETE，物件獲得 PK，但不提交交易。
- `commit()`：flush + 提交交易，使改變永久。
- `rollback()`：撤銷所有改變，回到上次 commit 狀態。
- `expunge_all()`：清除 session 中的所有物件，但不提交。

**在測試中的應用：**

```python
def test_task_with_explicit_session_commit(app):
    """
    示範：何時必須 commit，何時只需 flush。
    """
    user = User(name="Alice", email="alice@example.com", password="hashed")
    db.session.add(user)
    db.session.flush()  # 此時 user.id 已分配，但交易未提交
    
    # 在同一個 session 內可以建立依賴此 user 的任務
    task = Task(user_id=user.id, title="Do something")
    db.session.add(task)
    db.session.flush()
    
    assert task.id is not None  # FK 與主鍵都有了
    
    # 必須 commit，才能讓其他 session 或離開 app context 後看到這筆資料
    db.session.commit()
    
    # 現在可以在新的 session 中查詢
    # （雖然測試通常在同一個 session 中進行）
    fetched = db.session.get(Task, task.id)
    assert fetched.title == "Do something"
```

#### 6.1.2 Transactional 隔離與測試資料污染

**問題：** 如果測試不清除資料，後續測試會看到前面測試留下的污染。

**解決方案 1：Teardown 清理（目前做法）**

```python
@pytest.fixture
def app():
    """每個測試後清除資料庫。"""
    app = Flask(__name__)
    app.config["SQLALCHEMY_DATABASE_URI"] = "sqlite:///:memory:"
    db.init_app(app)
    
    with app.app_context():
        db.create_all()
        yield app
        db.session.remove()  # 清除 session
        db.drop_all()        # 清除表格
```

**解決方案 2：Rollback 隔離（更高級）**

某些專案使用 transaction-per-test，在測試後 rollback 交易（不清表，重置為測試前狀態）：

```python
@pytest.fixture
def app_with_rollback():
    """使用交易 rollback 隔離每個測試。"""
    app = Flask(__name__)
    app.config["SQLALCHEMY_DATABASE_URI"] = "sqlite:///:memory:"
    db.init_app(app)
    
    with app.app_context():
        db.create_all()
        connection = db.engine.connect()
        transaction = connection.begin()
        session = db.sessionmaker(bind=connection)()
        
        yield app
        
        session.close()
        transaction.rollback()
        connection.close()
```

此方法的優點：
- 更快（不需重建表），適合大量測試。
- 確保完全隔離。

缺點：
- 某些資料庫操作（如 DDL）無法在 rollback 中進行；SQLite in-memory 內部有限制。
- 複雜度高。

在本專案中使用 **drop_all** 方案因為簡單且足夠快。

#### 6.1.3 Lazy Loading 與 Eager Loading 陷阱

**問題：** Session 結束後無法存取 lazy-loaded 關聯。

```python
# 危險程式碼（產生 DetachedInstanceError）
user = db.session.get(User, 1)
db.session.close()  # Session 結束

# 此時存取 user.tasks 會失敗，因為 tasks 是 lazy-loaded
print(user.tasks)  # ❌ DetachedInstanceError
```

**解決方案：**

- 在 session 內 eager load：
  ```python
  from sqlalchemy.orm import joinedload
  user = db.session.query(User).options(joinedload(User.tasks)).get(1)
  db.session.close()
  print(user.tasks)  # ✅ OK
  ```

- 或在 service 層進行序列化：
  ```python
  def user_to_dict(user):
      return {
          'id': user.id,
          'name': user.name,
          'tasks': [task.id for task in user.tasks]  # 在 session 內完成
      }
  ```

### 6.2 Service 層測試的完整範例

#### 6.2.1 設計一個易測試的 Service

**原則：**
1. Service 函式應該是純粹的商業邏輯；不應包含 Flask context、HTTP 細節。
2. Service 應接受所有必要的參數（例如 user_id），而不依賴 global context（flask.g）。
3. Service 應使用自定義的 exception 來表示錯誤（帶 status_code），便於 blueprint 層統一映射。

**範例：Task Service**

```python
# services/task_service.py
from models.task import Task
from models.user import User
from models import db

class TaskOperationError(Exception):
    def __init__(self, message, status_code=400):
        super().__init__(message)
        self.status_code = status_code

def validate_task_data(data):
    """驗證 task payload 是否有未知欄位。"""
    allowed_fields = {'title', 'description', 'priority', 'deadline', 'status'}
    unknown = set(data.keys()) - allowed_fields
    if unknown:
        raise TaskOperationError(f"Unknown fields: {unknown}", 400)

def create_task_for_user(user_id: int, task_data: dict) -> int:
    """
    為使用者建立任務。
    
    參數：
        user_id: 使用者 ID
        task_data: 包含 title, description 等的字典
    
    回傳：
        建立的任務 ID
    
    異常：
        TaskOperationError: 若 user 不存在或資料無效
    """
    user = db.session.get(User, user_id)
    if not user:
        raise TaskOperationError(f"User {user_id} not found", 404)
    
    validate_task_data(task_data)
    
    task = Task(
        user_id=user_id,
        title=task_data.get('title'),
        description=task_data.get('description'),
        priority=task_data.get('priority', 3),
        deadline=task_data.get('deadline'),
    )
    db.session.add(task)
    db.session.commit()
    return task.id

def update_task_for_user(task_id: int, user_id: int, updates: dict):
    """更新任務，檢查所有權。"""
    task = db.session.get(Task, task_id)
    if not task:
        raise TaskOperationError(f"Task {task_id} not found", 404)
    if task.user_id != user_id:
        raise TaskOperationError("Not your task", 403)
    
    validate_task_data(updates)
    
    for key, value in updates.items():
        if hasattr(task, key):
            setattr(task, key, value)
    db.session.commit()
```

### 6.3 Blueprint 層測試的完整範例

#### 6.3.1 設計 Blueprint 測試的三個層次

```python
def login():
```

### 7.4 Store / State Management 測試（Pinia 範例）

#### 7.4.1 簡單 Store 測試

```typescript
// src/stores/taskStore.ts
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'
import { useTaskApi } from '@/composables/useTaskApi'

export const useTaskStore = defineStore('task', () => {
  const tasks = ref([])
  const loading = ref(false)
  const error = ref(null)
  const { getTaskList, createTask } = useTaskApi()

  const totalTasks = computed(() => tasks.value.length)
  const completedCount = computed(() =>
    tasks.value.filter((t) => t.completed).length
  )

  async function fetchTasks() {
    loading.value = true
    error.value = null
    try {
      const data = await getTaskList()
      tasks.value = data || []
    } catch (err) {
      error.value = err.message
    } finally {
      loading.value = false
    }
  }

  async function addTask(title, description) {
    try {
      const newTask = await createTask({ title, description })
      tasks.value.push(newTask)
      return newTask
    } catch (err) {
      error.value = err.message
      throw err
    }
  }

  return {
    tasks,
    loading,
    error,
    totalTasks,
    completedCount,
    fetchTasks,
    addTask,
  }
})
```

## 8. 面試問答與答案範例

### 8.1 後端測試相關面試題

**Q1：為什麼要使用 in-memory SQLite 進行測試，而不是 Postgres？**

答：
- 速度：in-memory SQLite 極快，測試可在毫秒內完成。
- 隔離：每個測試可獨立建立/清除資料庫，無污染風險。
- 環境簡單：無需啟動外部 Postgres，本機測試無依賴。

缺點是 SQLite 與 Postgres 在 SQL dialect、交易、某些特性（如 JSONB）上有差異。
解決方案是在 CI 中除了使用 SQLite 快速迴歸測試外，也定期跑一次整合測試使用真實 Postgres。

**Q2：在 service 層測試中，為什麼需要 `db.session.commit()`？不能只 `flush()` 嗎？**

答：
- `flush()` 只是生成 SQL 並執行，分配主鍵，但不提交交易。
- `commit()` 才會真正提交交易，使資料永久化。
- 如果測試中建立 user 後只 flush 但不 commit，然後新建一個 task 引用 user.id，可能會因為 user 尚未完全持久化導致外鍵約束失敗或其他問題。
- 在同一個 session 內，flush 可能足夠；但為保險起見且便於測試隔離，建議明確 commit。

**Q3：如何測試 service 層的權限邏輯（例如使用者 A 無法修改使用者 B 的任務）？**

答：
```python
def test_update_task_permission_denied(app):
    user_a = create_user(...)
    user_b = create_user(...)
    
    task = create_task_for_user(user_a.id, {'title': 'A Task'})
    
    with pytest.raises(TaskOperationError) as exc:
        update_task_for_user(task.id, user_b.id, {'title': 'Hacked'})
    
    assert exc.value.status_code == 403
```

**Q4：什麼時候應該使用 monkeypatch，什麼時候應該用 mock？**

答：
- `monkeypatch`（pytest 內置）：臨時替換函式、環境變數、屬性。簡潔，特別適合簡單替換。
- `unittest.mock` / `pytest-mock`：更複雜的 mocking，例如追蹤呼叫次數、設定副作用、return values。

實務建議：簡單情況用 monkeypatch；複雜情況（需追蹤、多次呼叫）用 mock。

**Q5：如何在 blueprint 層測試中驗證 JWT token 的正確性？**

## 9. 常見陷阱與最佳實務清單

### 9.1 後端測試常見陷阱

1. **隔離性破裂：** 測試在全域 session 中操作，導致污染。
   - 解決：確保 conftest 的 app fixture 每次 drop_all()。

2. **忘記 commit：** 建立測試資料但不 commit，導致後續查詢無法看到。
   - 解決：明確 db.session.commit()。

3. **Mock 過度：** 為了測試而 mock 掉所有東西，最後測試無法驗證真實邏輯。
   - 解決：只 mock 外部服務（AI, 支付），保留資料庫。

4. **缺少錯誤路徑測試：** 只測試 happy path，忽略 400 / 403 / 404。
   - 解決：每個端點至少測 1 正 2 負。

5. **依賴測試執行順序：** 測試 A 建立資料，測試 B 依賴此資料。
   - 解決：每個測試獨立，fixture 在每個測試中獨立建立。

### 9.2 前端測試常見陷阱

1. **過度 mock：** Mock 掉太多，導致測試無法發現真實問題。
   - 解決：只 mock 外部 API（MSW），保留元件邏輯。

2. **非同步混亂：** 不等待 promise，測試立即完成，斷言未執行。
   - 解決：使用 `await`, `waitFor`, `nextTick()`。

3. **時間依賴：** 測試依賴 setTimeout，不穩定。
   - 解決：使用 `vi.useFakeTimers()` 或 `vi.waitFor()`。

4. **缺少 test-id：** 靠 CSS class 或文字選擇，脆弱。
   - 解決：加入 `data-testid` 屬性。

5. **E2E 過度：** 把所有測試寫成 E2E，執行慢。
   - 解決：遵循金字塔模型，多單元少 E2E。

### 9.3 最佳實務清單

#### 後端

- [ ] 每個 service 函式有對應的測試，涵蓋 happy path 和錯誤路徑。
- [ ] Blueprint 層測試涵蓋驗證、授權、錯誤映射。
- [ ] Mock 外部服務（AI, 支付, 郵件）以加快測試。
- [ ] 使用 in-memory SQLite 進行快速回歸，CI 中偶爾跑真實 Postgres 測試。
- [ ] 測試覆蓋率 > 80%，關鍵路徑 100%。
- [ ] 每個 blueprint 端點至少測 1 正 2 負（400, 403, 404 等）。

#### 前端

- [ ] 每個 component 有單元測試，使用 @testing-library 或 Vue Test Utils。
- [ ] 複雜邏輯（composables, stores）有獨立測試。
- [ ] 使用 MSW 模擬 API，避免呼叫真實後端。
- [ ] 關鍵使用者流程有 E2E 測試（登入、建立資源等）。
- [ ] 測試覆蓋率 > 70%。

#### CI/CD

- [ ] 在 PR 上執行快速測試（單元 + 簡短整合）。
- [ ] 夜間執行完整套件（含 E2E）。
- [ ] 覆蓋率降低時失敗（--coverage-threshold）。

## 14. 實戰案例：完整測試一個複雜功能

### 14.1 案例：「建立任務並分配成員」流程

**業務需求：**
1. 使用者 A 建立任務 T。
2. A 將 B、C 加入 T（add members）。
3. A 設定 B 為 editor，C 為 viewer。
4. B 可編輯任務，C 只能檢視。
5. 若移除 B，B 再無法存取 T。

**設計測試覆蓋：**

#### 14.1.1 Service 層測試

```python
# tests/services/test_task_member_service.py
import pytest
from models import db
from models.user import User
from models.task import Task, TaskMember
from services.task_service import (
    create_task_for_user,
    add_task_member,
    update_task_member_role,
    remove_task_member,
    TaskOperationError,
)

def test_add_member_to_task_happy_path(app):
    """正常流程：建立任務，加入成員，驗證角色。"""
    owner = User(name='Alice', email='alice@example.com', username='alice', password='h')
    member = User(name='Bob', email='bob@example.com', username='bob', password='h')
    db.session.add_all([owner, member])
    db.session.commit()
    
    task_id = create_task_for_user(owner.id, {'title': 'Collab Task'})
    
    add_task_member(task_id, owner.id, member.id, 'editor')
    
    member_record = db.session.query(TaskMember).filter_by(
        task_id=task_id, user_id=member.id
    ).first()
    assert member_record is not None
    assert member_record.role == 'editor'

def test_add_member_already_exists(app):
    """邊界情況：成員已存在，應拒絕重複加入。"""
    owner = User(name='Alice', email='alice@example.com', username='alice', password='h')
    member = User(name='Bob', email='bob@example.com', username='bob', password='h')
    db.session.add_all([owner, member])
    db.session.commit()
    
    task_id = create_task_for_user(owner.id, {'title': 'Task'})
    add_task_member(task_id, owner.id, member.id, 'editor')
    
    with pytest.raises(TaskOperationError) as exc:
        add_task_member(task_id, owner.id, member.id, 'viewer')
    assert exc.value.status_code == 409  # Conflict

def test_update_member_role(app):
    """成員角色更新：editor → viewer。"""
    owner = User(name='Alice', email='alice@example.com', username='alice', password='h')
    member = User(name='Bob', email='bob@example.com', username='bob', password='h')
    db.session.add_all([owner, member])
    db.session.commit()
    
    task_id = create_task_for_user(owner.id, {'title': 'Task'})
    add_task_member(task_id, owner.id, member.id, 'editor')
    
    update_task_member_role(task_id, owner.id, member.id, 'viewer')
    
    member_record = db.session.query(TaskMember).filter_by(
        task_id=task_id, user_id=member.id
    ).first()
    assert member_record.role == 'viewer'

def test_remove_member_revokes_access(app):
```

## 16. 故障排除與除錯技巧

### 16.1 常見測試失敗與解決方案

| 問題 | 症狀 | 原因 | 解決方案 |
|------|------|------|--------|
| DetachedInstanceError | 在 session 外存取 lazy-loaded 屬性 | Session 已關閉 | 在 session 內 eager load 或序列化 |
| Foreign Key Constraint Failed | 插入資料時外鍵錯誤 | FK 值不存在或未 commit | 確保 FK 參考資料已 commit |
| Test 順序依賴 | 某個測試失敗時其他測試也失敗 | 測試間污染 | 使用獨立 fixture，避免全域狀態 |
| 隨機失敗（Flaky tests） | 間歇性失敗 | 非 determinism（時間、隨機值） | Mock 時間，使用固定種子 |
| 超時 | 測試卡住 | 無限迴圈或外部 I/O 阻塞 | 加 timeout，mock 外部呼叫 |

## 18. 測試最佳實務檢查清單

使用此清單進行自評與 code review：

### 18.1 測試設計清單

- [ ] 每個函式有對應的測試（至少 happy path）

- [ ] 邊界值測試（空、null、超大值）

## 20. 測試工具生態與選型指南

### 20.1 後端測試工具對比

| 工具 | 使用場景 | 優點 | 缺點 | 相似工具 |
|------|--------|------|------|---------|
| **pytest** | Python 單元/整合測試 | 簡潔、功能強、plugin 豐富 | 學習曲線陡 | unittest, nose2 |
| **unittest** | Python 標準庫測試 | 內置、OOP 設計 | 冗長、功能少 | pytest, nose2 |
| **Mock / monkeypatch** | Stub 依賴 | 內置或 pytest fixture | 需理解如何使用 | unittest.mock, pytest-mock |
| **pytest-cov** | 覆蓋率報告 | 整合式、HTML 報告 | 不檢查測試品質 | coverage, codecov |
| **pytest-xdist** | 平行執行 | 大幅加快 | 需要注意隔離性 | tox, nox |
| **hypothesis** | 屬性驅動測試 | 自動生成測試案例 | 學習成本高 | QuickCheck (Haskell) |
| **locust** | 負載/壓力測試 | 易用、視覺化 | 需要額外搭建 | k6, Apache JMeter |

### 20.2 前端測試工具對比

| 工具 | 使用場景 | 優點 | 缺點 | 相似工具 |
|------|--------|------|------|---------|
| **Vitest** | Vue/React 單元測試 | 快速、Vite 集成 | 生態較新 | Jest, Mocha |
| **Jest** | JavaScript 單元測試 | 標準化、社區大 | 重、較慢 | Vitest, Mocha |
| **@testing-library** | 元件測試 | 聚焦使用者行為 | 學習曲線陡 | Enzyme, React Testing Library |
| **MSW** | API Mock | 同步本地與 E2E | 需額外學習 | nock, axios-mock-adapter |
| **Playwright** | E2E 測試 | 跨瀏覽器、快速 | 測試較慢 | Cypress, Selenium |
| **Cypress** | E2E 測試 | 開發友善、調試好 | 僅 Chromium/Firefox | Playwright, Selenium |

**選型建議：**
- **小型專案**：pytest + Vitest + Playwright
- **大型專案**：pytest + pytest-cov + pytest-xdist + Vitest + MSW + Playwright
- **高性能要求**：加上 locust/k6 進行負載測試

## 26. 完整的面試會話模擬

### 26.1 場景：中階後端工程師面試

**面試官：** 你之前提到測試，能否描述一個你實現過的複雜測試場景？

**你的回答（預期結構）：**

「我在做一個任務管理系統的時候，遇到一個需求：使用者可以建立任務、邀請成員、設定不同角色（owner/editor/viewer），並且要驗證權限。

**測試挑戰是：** 
- 需要驗證 24 種權限組合（4 角色 × 6 操作）
- 需要驗證成員移除後會立即失去存取權
- 需要驗證 API 層級的權限映射（例如 403 Forbidden）

**我的解決方案是分層：**

1. **Service 層（單元測試）**：直接測試業務邏輯
   - `add_task_member(task_id, user_id, role)` 是否正確儲存
   - `can_user_perform_action(user_id, task_id, action)` 是否返回正確權限
   - 使用 pytest 參數化測試 24 種組合，避免重複程式碼

2. **Blueprint 層（整合測試）**：測試 HTTP 層面
   - 驗證無 token 返回 401
   - 驗證無權限返回 403
   - 驗證成功返回 200/201
   - 測試 token 過期情況

3. **E2E 層（使用者流程）**：測試完整工作流
   - Alice 建立任務 → 邀請 Bob → Bob 編輯 → Alice 移除 Bob → Bob 無法存取

**工程實踐：**
- 使用 in-memory SQLite 確保速度
- Mock 外部依賴（如 AI provider）
- 在 CI 中平行執行測試，節省時間
- 覆蓋率目標 80%+

最終結果是一個可靠的測試套件，每當程式碼改變時都能快速捕捉回歸。」

---

**面試官：** 你如何處理測試中的非確定性（flaky tests）？

**你的回答：**

「Flaky tests 是常見問題，我的做法是：

1. **識別根本原因：**
   - 系統時間依賴（時區、時間戳記）→ mock 時間
   - 隨機值（UUID、隨機排序） → mock random / 使用固定種子
   - 外部 API（不穩定） → mock 或使用 MSW
   - 並發問題 → 加鎖或使用 lock-based fixtures

2. **例子：我之前遇到的一個 test**
   - 測試任務的 created_at 時間戳記
   - 問題：不同執行環境、不同時區會產生不同結果
   - 解決：使用 freezegun mock 系統時間
   
   ```python
   def test_task_created_at_timestamp(app, freezegun):
       freezegun.freeze_time('2026-05-01')
       task = create_task(...)
       assert task.created_at == datetime(2026, 5, 1)
   ```

3. **預防措施：**
   - Code review 時關注 test 是否依賴外部資源
   - 在 CI 中重複執行 flaky test 幾次以捕捉問題
   - 用 pytest-rerunfailures 標記可能 flaky 的 test，自動重試」

---

**面試官：** 覆蓋率應該是多少？怎樣避免過度測試？

**你的回答：**

「覆蓋率不是終極目標，而是指標。我的做法是：

**覆蓋率目標（分級）：**
- **100%**：關鍵業務邏輯（登入、支付、權限檢查）
- **80-90%**：核心 service 函式
- **60-70%**：輔助函式、工具類
- **不需要**：簡單 getter、框架程式碼

**避免過度測試：**
- 不測試框架（Flask、SQLAlchemy 內部）
- 不測試簡單 getter/setter（除非有驗證邏輯）
- 不測試與外部系統的簡單包裝（例如直接呼叫第三方 API）

**例子：**
```python
# ❌ 過度測試（不值得）
def test_user_get_name():
    user = User(name='Alice')
    assert user.name == 'Alice'  # 測試 getter，無邏輯

# ✅ 值得測試（含驗證邏輯）
def test_user_set_email_validates():
    user = User()
    with pytest.raises(ValueError):
        user.email = 'invalid-email'  # 測試驗證
```

實務中我用 coverage 工具監視，設定門檻為 75%，超過沒有獎勵，低於則 CI 失敗。」

```
### 31.1 高級後端面試題

**Q：你如何測試一個涉及多個微服務的複雜流程？**

答：
```
設想場景：
1. 使用者服務（User Service）
2. 任務服務（Task Service）
3. 通知服務（Notification Service）

流程：User A 建立任務 → Task Service 建立任務 → Notification Service 發送通知 → User B 收到通知

測試策略分層：

1. **單服務層級（Service）**
   - 每個服務單獨測試（mock 其他服務）
   - 驗證業務邏輯正確性
   - 覆蓋率目標 ≥ 80%

2. **服務整合層級（API Gateway / Contract Testing）**
   - 使用 Pact 定義服務間契約
   - User Service 模擬 Task Service 的 response
   - Task Service 模擬 Notification Service 的 response
   - 驗證序列化與反序列化正確

3. **完整整合層級（Test Environment）**
   - 啟動所有微服務（Docker Compose）
   - 執行完整流程
   - 驗證最終結果（User B 收到通知）
   - 這一層較慢，通常在 CI 中每天或每小時執行一次

4. **E2E 層級（Real Environment）**
   - Staging 環境中的真實測試
   - 驗證部署與運營方面（監控、日誌）

測試金字塔模型：
```
       E2E (1%)
      /  \
   整合 (5%)
   /       \
契約 (15%)
/           \
服務 (79%)
```

```python
import pytest
import threading
import time

def test_transaction_isolation(app):
    """驗證交易隔離級別。"""
    user = create_user(...)
    
    # Session 1：開始交易，更新 user
    session1 = db.sessionmaker()
    session1.begin()
    user1 = session1.query(User).filter_by(id=user.id).first()
    user1.balance = 1000
    session1.flush()  # 還未 commit
    
    # Session 2：在 session 1 未提交時查詢
    session2 = db.sessionmaker()
    user2 = session2.query(User).filter_by(id=user.id).first()
    
    # 在 REPEATABLE_READ 隔離下，user2.balance 應該看不到更新
    assert user2.balance != 1000  # 或應該是 1000，取決於隔離級別
    
    # 提交 session 1
    session1.commit()
    
    # 重新查詢
    user2_refreshed = session2.query(User).filter_by(id=user.id).first()
    assert user2_refreshed.balance == 1000  # 現在可以看到

def test_deadlock_recovery(app):
    """驗證死鎖恢復。"""
    user_a, user_b = create_user(...), create_user(...)
    
    def transfer_a_to_b():
        try:
            transfer_money(user_a.id, user_b.id, 100)
        except DeadlockError:
            # 應用層應重試
            transfer_money(user_a.id, user_b.id, 100)
    
    def transfer_b_to_a():
        try:
            transfer_money(user_b.id, user_a.id, 50)
        except DeadlockError:
            transfer_money(user_b.id, user_a.id, 50)
    
    # 並發執行，可能導致死鎖
    t1 = threading.Thread(target=transfer_a_to_b)
    t2 = threading.Thread(target=transfer_b_to_a)
    
    t1.start()
    t2.start()
    t1.join()
    t2.join()
    
    # 驗證最終狀態一致
    assert user_a.balance + user_b.balance == initial_total
```

**Q：如何測試 Vue/React 中的複雜狀態管理（Pinia / Redux）流程？**

```typescript
  beforeEach(() => {
    setActivePinia(createPinia())
  })

    const store = useTaskStore()
    
    // 初始狀態
    expect(store.loading).toBe(false)
    expect(store.error).toBe(null)

    const promise = store.createTask({ title: 'New Task' })

    expect(store.loading).toBe(false)

  })

  it('handles API error and retries', async () => {
    const store = useTaskStore()
    
    // 首次呼叫失敗
    let callCount = 0
    const mockFetch = vi.fn(async () => {
      callCount++
      if (callCount === 1) {
        throw new Error('Network error')
      }
      return { id: 1, title: 'Task' }
    })
    
    store.createTaskWithRetry = async (data) => {
      for (let i = 0; i < 3; i++) {
        try {
          const task = await mockFetch(data)
          store.tasks.push(task)
          return task
        } catch (err) {
          if (i === 2) throw err
          await new Promise(r => setTimeout(r, 100))  // 延遲後重試
        }
      }
    }
    
    const result = await store.createTaskWithRetry({ title: 'Task' })
    
    expect(result).toBeDefined()
    expect(mockFetch).toHaveBeenCalledTimes(2)
  })

  it('handles concurrent requests correctly', async () => {
    const store = useTaskStore()
```

## 附錄 B：測試速查表（Cheat Sheet）

### Python / pytest

```python
# 基本結構
def test_something(app, monkeypatch):
    # Arrange
    obj = create_object()
    
    # Act
    result = obj.do_something()
    
    # Assert
    assert result == expected

# 異常測試
with pytest.raises(ValueError) as exc_info:
    some_function_that_raises()
assert "error message" in str(exc_info.value)

# 參數化
@pytest.mark.parametrize('input,expected', [
    (1, 2),
    (2, 4),
])
def test_something(input, expected):
    assert double(input) == expected

# Fixture
@pytest.fixture
def user():
    user = User(name='Test')
    db.session.add(user)
    db.session.commit()
    yield user
    db.session.delete(user)
    db.session.commit()

# Mock
monkeypatch.setattr('module.function', MagicMock(return_value=42))

# 標記
@pytest.mark.slow
@pytest.mark.skip(reason="Not implemented")
def test_something():
    pass
```

### TypeScript / Vitest

```typescript
// 基本結構
import { test, expect, describe } from 'vitest'

describe('Component', () => {
  test('should render title', () => {
    // Arrange
    const props = { title: 'Hello' }
    
    // Act
    const wrapper = render(Component, { props })
    
    // Assert
    expect(wrapper.text()).toContain('Hello')
  })
})

// 非同步測試
test('fetches data', async () => {
  const data = await fetchData()
  expect(data).toBeDefined()
})

// Mock
vi.mock('@/api', () => ({
  fetchUser: vi.fn(() => ({ id: 1, name: 'Alice' }))
}))

// Spy
const spy = vi.spyOn(module, 'function')
spy.mockReturnValue(42)
expect(spy).toHaveBeenCalled()

// 參數化
describe.each([
  [1, 2],
  [2, 4],
])('double', (input, expected) => {
  test(`double(${input}) === ${expected}`, () => {
    expect(double(input)).toBe(expected)
  })
})
```
