## 1) LlamaIndex 的定位與核心問題

LlamaIndex 的核心任務是：
把「原始資料」轉成「可被 LLM 有效檢索與回答的知識系統」。

它重點解決的不是模型推理本身，而是資料側問題：

1. 資料怎麼切。
2. 索引怎麼建。
3. 查詢怎麼找。
4. 回答怎麼合成。

可把它視為 RAG 的資料治理層。

---

## 2) 資料生命週期：從文件到可回答知識

在教學上，最重要的是先建立完整流程觀：

1. 接入資料
    - 把檔案、網頁、資料庫內容讀進系統。
2. 切分資料
    - 將長文本分割成可檢索單元。
3. 表徵資料
    - 將內容轉成可比對的語意表示。
4. 建立索引
    - 把可檢索單元組成高效率查詢結構。
5. 查詢與召回
    - 依問題找回最相關片段。
6. 回答合成
    - 將片段整理成可讀答案。

這條生命週期決定了 RAG 的上限。

---

## 3) 核心元件總覽

| 元件 | 在系統中的位置 | 核心職責 |
|---|---|---|
| Document | 資料入口 | 承載原始內容與基礎 metadata |
| Node | 檢索最小單位 | 將文件拆成可召回片段 |
| Index | 查詢結構 | 組織 Node 以提升查詢效率與準確性 |
| Retriever | 召回器 | 根據問題挑出候選片段 |
| Query Engine | 對外入口 | 封裝召回與回答流程 |
| Response Synthesizer | 回答整合器 | 把多片段組成連貫答案 |

重點是分工：
LlamaIndex 用清晰元件邊界，把資料處理從黑箱改成可調整流程。

---

## 4) 索引思維：不是只有向量，還有結構選擇

很多人把 LlamaIndex 等同於向量索引，但本質上它是索引框架。

你需要理解的是「索引是策略選擇」：

1. 語意型索引
    - 適合自然語言問答與語意召回。
2. 結構型索引
    - 適合摘要、分層導航、主題化瀏覽。
3. 混合型索引
    - 同時考慮語意與關鍵字，提升穩定性。

選擇哪種索引，本質上是在選擇「你希望系統怎麼找答案」。

---

## 5) 檢索策略：召回、過濾、重排序

高品質檢索通常由三段組成：

1. 召回
    - 先找出一批可能相關片段。
2. 過濾
    - 用 metadata、主題、時間範圍等條件剔除噪音。
3. 重排序
    - 對候選片段再評分，讓最相關內容排前面。

教學上可記一個原則：
先求召回完整，再求排序精準。

---

## 6) 回答合成：如何從片段到完整答案

檢索只是找到材料，回答品質取決於合成策略。

常見合成思路：

1. 直接整合
    - 把候選片段合併後生成答案。
2. 分步整合
    - 先各片段提取重點，再統一彙總。
3. 引用約束
    - 要求每個關鍵陳述可對應到來源片段。

一個成熟的 RAG 系統，重點不是「有沒有答案」，而是「答案是否可追溯」。

---

## 7) 影響品質的關鍵槓桿

以下四項最影響最終效果：

1. 切塊策略
    - 太大會混雜，太小會破碎。
2. metadata 品質
    - 標籤不準確時，檢索會出現跨主題汙染。
3. 召回範圍
    - 範圍太窄會漏資訊，太寬會引入噪音。
4. 回答約束
    - 缺乏引用與邊界時，容易產生幻覺式補全。

這些通常比更換大模型更值得優先優化。

---

## 8) 與 LangChain、LangGraph 的關係

可以用一句話記住分工：

1. LlamaIndex 管資料與檢索。
2. LangChain 管能力編排。
3. LangGraph 管多步流程控制。

三者不是替代關係，而是可疊加的系統分層。

---

## 9) 常見誤解與修正

| 常見誤解 | 實際情況 | 修正觀點 |
|---|---|---|
| 用 LlamaIndex 就會自動高準確 | 準確度取決於資料治理與檢索策略 | 框架只是手段，關鍵是資料品質 |
| 只要向量檢索就夠了 | 不同場景需要不同索引結構 | 依問題型態選索引，而非固定單一解 |
| 回答好不好只看模型 | 召回內容決定回答上限 | 先保證召回質量，再談生成模型 |
| 有檢索就不會幻覺 | 無約束合成仍可能偏離來源 | 要加來源引用與回答邊界規則 |

---

## 10) 一頁式總結

1. LlamaIndex 的本質是 RAG 資料層工程。
2. 核心價值在於把資料處理拆成可管理、可優化的流程。
3. 若你想穩定提升問答品質，優先關注切塊、檢索與合成策略。
4. 真正成熟的系統不是只會回答，而是能提供可追溯、可維護的答案。
