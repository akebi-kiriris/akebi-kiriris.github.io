## 前言

### 測試驅動開發 (TDD)、領域驅動開發 (DDD) 和事件驅動架構

依知名度排序，我們知道有三種應對複雜性的工具：

1. 測試驅動開發(TDD) 可協助我們編寫正確的程式碼，並使我們能夠重構或新增功能，而無需擔心迴歸問題。但是，要充分發揮測試的作用並非易事：如何確保測試運行速度盡可能快？如何從快速、無依賴的單元測試中獲得盡可能多的程式碼覆蓋率和回饋，同時盡可能減少速度較慢、不穩定的端到端測試的數量？

2. 領域驅動設計(DDD) 要求我們集中精力建立一個良好的業務領域模型，但我們如何確保我們的模型不受基礎設施問題的困擾，並且不會變得難以更改？

3. 透過訊息整合的鬆散耦合（微）服務（有時稱為 響應式微服務）是管理跨多個應用程式或業務領域複雜性的成熟解決方案。但是，如何使它們與 Python 世界現有的工具（例如 Flask、Django、Celery 等）相容並不總是顯而易見的。

## 介紹

軟體系統容易陷入混亂。

當我們最初建立一個新系統時，我們雄心勃勃地設想程式碼會簡潔有序，但隨著時間的推移，我們發現程式碼中累積了大量冗餘程式碼和邊界情況，最終變成了一團混亂的管理類別和工具模組。

我們發現，我們精心設計的分層架構就像一塊過度濕潤的甜點一樣坍塌了。

混亂的軟體系統的特徵是功能同質化：
- API 處理程序擁有領域知識，卻負責發送電子郵件和日誌記錄；
- 「業務邏輯」類別不進行任何計算，卻執行 I/O 操作；
- 所有元件都相互耦合，以至於更改系統的任何部分都充滿風險。

>確是如此，一開始做專案只有app.py和blueprint(當然當初我連分層的觀念都沒有)，混亂程度可想而知。


### 封裝與抽象 (Encapsulation and Abstractions)
**封裝**和**抽象化**是我們程式設計師都會本能地使用的工具，即使我們可能不會直接使用這些字詞。

封裝一詞涵蓋兩個密切相關的概念：**簡化行為**和**隱藏資料**。本文主要討論第一種意義。我們透過識別程式碼中需要完成的任務，並將該任務交給一個定義明確的物件或函數來封裝行為。我們將這個物件或函數稱為抽象。

請看以下兩段Python程式碼：

```python
import json
from urllib.request import urlopen
from urllib.parse import urlencode

params = dict(q='Sausages', format='json')
handle = urlopen('http://api.duckduckgo.com' + '?' + urlencode(params))
raw_text = handle.read().decode('utf8')
parsed = json.loads(raw_text)

results = parsed['RelatedTopics']
for r in results:
    if 'Text' in r:
        print(r['FirstURL'] + ' - ' + r['Text'])
```

```python
import requests

params = dict(q='Sausages', format='json')
parsed = requests.get('http://api.duckduckgo.com/', params=params).json()

results = parsed['RelatedTopics']
for r in results:
    if 'Text' in r:
        print(r['FirstURL'] + ' - ' + r['Text'])
```        


兩段程式碼的功能相同：它們都將表單編碼後的值提交到 URL，以便呼叫搜尋引擎 API。但第二段程式碼更易於閱讀和理解，因為它採用了更高的抽象層次。

我們可以更進一步，明確並命名我們希望程式碼執行的任務，並使用更高層次的抽象來使其更加明確：


```python
import duckduckpy
for r in duckduckpy.query('Sausages').related_topics:
    print(r.first_url, ' - ', r.text)
```

利用抽象來封裝行為是一種強大的工具，可以使程式碼更具表達力、更易於測試和更易於維護。


### 分層 (Layering)

封裝和抽象化透過隱藏細節和保護資料一致性來幫助我們，但我們也需要關注物件和函數之間的交互作用。

當一個函數、模組或物件使用另一個時，我們就說它們之間存在依賴關係。這些依賴關係構成了一種網絡或圖。

就像一團亂麻，依賴關係完全失控。修改圖中的一個節點會變得非常困難，因為它可能會影響系統的許多其他部分。

分層架構是解決這個問題的一種方法。在分層架構中，我們將程式碼劃分為不同的類別或角色，並制定規則來規定哪些類別的程式碼可以相互呼叫。

最常見的例子之一是分層架構中所示的三層架構。

![image](https://www.cosmicpython.com/book/images/apwp_0002.png)

>我的prajekt目前採用的也是這種 : blueprint、service、repository

分層架構或許是建構業務軟體最常見的模式。在這個模型中，我們有使用者介面元件（可以是網頁、API 或命令列）；這些使用者介面元件與業務邏輯層通信，業務邏輯層包含我們的業務規則和工作流程；最後，我們還有一個資料庫層，負責儲存和檢索資料。

在本書的剩餘部分，我們將遵循一個簡單的原則，有系統地徹底改變這個模型。


### 依賴倒置原理 (The Dependency Inversion Principle, DIP)

DIP的正式定義：
- 高層模組不應該依賴底層模組。兩者都應該依賴抽象概念。
- 抽象概念不應依賴細節，相反，細節應該依賴抽象概念。

高階模組是組織真正關心的程式碼。例如，您可能在一家製藥公司工作，您的高級模組負責處理患者和臨床試驗；或者您在銀行工作，您的高級模組負責管理交易和兌換。

軟體系統的高階模組就是那些處理我們現實世界概念的函數、類別和套件。

相比之下，底層模組是組織並不關心的程式碼。人力資源部門不太可能對檔案系統或網路套接字感興趣。
你也很少會和財務團隊討論 SMTP、HTTP 或 AMQP。對於非技術利害關係人來說，這些底層概念既不有趣也不相關。

他們只關心高層概念是否運作正常。如果工資發放按時完成，你的企業不太可能關心這是透過定時任務還是在 Kubernetes 上運行的臨時函數實現的。

依賴並不一定意味著導入或調用，而是一個更普遍的概念，`即一個模組了解或需要另一個模組`。

我們之前已經提到過抽象：它們是簡化的接口，封裝了行為，就像我們的 DuckDuckGo 模組封裝了搜尋引擎的 API 一樣。

```
計算機科學中的所有問題都可以透過增加一層間接定址來解決。
                                        ——大衛‧惠勒
```
>典中典的遇到問題就加中間層，之前B站的MCP相關影片中評論區看到的。當然他也有提到須避免over-engineering

因此，DIP 的第一部分指出，我們的業務程式碼不應該依賴技術細節；相反，兩者都應該使用抽象。

為什麼？簡而言之，因為我們希望能夠獨立地更改它們。高層模組應該能夠輕鬆地根據業務需求進行更改。

而底層模組（細節）在實踐中通常更難更改：想想看，重構函數名和定義、測試並部署資料庫遷移來更改列名之間的區別。我們不希望業務邏輯的變更因為與底層基礎設施細節緊密耦合而減慢速度。

但同樣重要的是，當需要更改基礎設施細節時（例如，對資料庫進行分片），無需更改業務層。在它們之間添加抽象層（即著名的間接層）可以讓兩者（更）獨立地進行更改。


### 所有業務邏輯的歸宿：領域模型 (The Domain Model)

但在徹底剖析我們的三層架構之前，我們需要更深入地探討中間層：高層模組或業務邏輯。設計失敗最常見的原因之一就是業務邏輯分散在應用程式的各個層級中，導致難以識別、理解和修改。

## 建構支援領域建模的架構


我們發現，許多開發人員在被要求設計新系統時，會立即著手建立資料庫模式，而將物件模型視為次要考量。

這正是問題開始出現的地方。正確的做法是，先考慮行為，並以此來驅動儲存需求。畢竟，我們的客戶並不關心資料模型，他們關心的是系統能做什麼；否則，他們直接使用電子表格就好了。

我們提出了四種關鍵設計模式：

- 倉庫模式(Repository pattern)是持久儲存概念的抽象
- 服務層模式(Service Layer)清楚地定義了我們的用例的起始點和結束點。
- 工作單元模式(Unit of Work pattern)提供原子操作
- 採用聚合模式(Aggregate pattern)來確保資料的完整性

### 什麼是領域模型？

在前言部分，我們使用「業務邏輯層」(business logic layer)來描述三層架構的中間層。本書後續章節將改用「領域模型」這個術語。領域模型是領域驅動設計（DDD）社群提出的術語，它能更好地表達我們想要表達的含義。

領域其實就是你試圖解決的問題的一種比較專業的說法。你的作者目前就職於一家線上家具零售商。根據你所討論的系統，領域可能是採購、產品設計，或是物流配送。大多數程式設計師每天的工作就是改善或自動化業務流程；領域指的是這些流程所支援的所有活動。

模型是對某個過程或現象的描述，它捕捉了該過程或現象的有用屬性。人類非常擅長在腦海中建構事物的模型。例如，當有人向你丟球時，你幾乎可以不假思索地預測它的運動軌跡，因為你心中有一個關於物體在空間中運動方式的模型。當然，你的模型並非完美無缺。人類對於物體在接近光速或真空中的運動方式缺乏直覺，因為我們的模型最初並非為涵蓋這些情況而設計的。但這並不意味著模型本身是錯誤的，而是意味著某些預測超出了模型的適用範圍。

領域模型是企業主對其業務的思維導圖。所有商業人士都有這種心智圖——這是人類思考複雜流程的方式。


#### 1. 模型是對現實的簡化
```
模型 ≠ 現實本身
```
只保留：
- 有用資訊
- 業務需要的資訊


#### 2. 每個模型都有適用範圍
```
超出範圍就會失效
```

例如我的prAjeKt(不是作者是我的)
```
Task
Timeline
Role
```
這些都是我對「專案管理」的模型

但：
- 不適合醫療系統
- 不適合銀行
- 不適合物流

#### 3. 領域語言會自然形成

這其實是：

**ubiquitous language**

DDD 超核心概念。
```
timeline
kanban
todo
viewer
editor
owner
```

這就是：
- domain language
- domain vocabulary


```
# 領域模型

- 是對現實業務的簡化
- 不追求完整真實，只保留有用資訊
- 每個模型都有適用範圍
- 商業術語會自然形成（ubiquitous language）
```
